# 技术选型与架构设计

> 创建日期：2026-06-05
> 目标项目：Happy Study — 程序员 AI 学习与面试平台

---

## 一、选型原则

| 原则 | 说明 |
|------|------|
| 🎯 **业务匹配优先** | 不追新，选最适合多 Agent 协同 + 实时交互场景的 |
| 💰 **成本可控** | 考虑到 LLM API 调用是主要成本，选型要考虑 token 消耗优化 |
| 🇨🇳 **国内友好** | 能在阿里云等国内环境顺畅部署，依赖国内可访问的镜像/源 |
| 🔧 **渐进增强** | MVP 先跑通核心流程，架构支持后续平滑扩展 |
| 🧩 **生态丰富** | 选社区活跃、文档齐全的技术，减少自己造轮子 |

## 二、各层选型结论

| 层级 | MVP 选型 | 核心理由 |
|------|---------|---------|
| 🚀 后端 | **Go 1.26 + Hertz + Eino** | 字节全家桶，Go 原生 Agent 编排 |
| 🎨 前端 | **Next.js + shadcn/ui** | 流式渲染强、交互式代码编辑器支持好 |
| 🗄️ 数据库 | **PostgreSQL + Pgvector** | MVP 一个库搞定关系+向量，省运维 |
| ⚡ 缓存 | **Redis** | 标配，无争议 |
| 🤖 主力模型 | **DeepSeek V3** | 性价比极高、国内直达、中文强 |
| 🧠 推理模型 | **DeepSeek R1** | 复杂诊断/代码审查用 |
| 📐 Embedding | **BGE-M3** | 多语言、检索强、可本地部署 |
| 📦 消息队列 | **NSQ / Redis Stream** | Go 原生、轻量 |
| 🏠 部署 | **Docker Compose (阿里云 ECS)** | MVP 够用 |
| 🛡️ 沙箱 | **Docker 容器** | 安全执行用户代码 |

## 三、架构关键决策

### 决策 1：单体还是微服务？

**MVP 阶段用单体**，在 Hertz 中用 group 做逻辑分层，后续可拆。

### 决策 2：实时通信方案

| 场景 | 方案 |
|------|------|
| Agent 回答流式输出 | SSE (Server-Sent Events) |
| 面试官实时反馈 | SSE |
| 交互式编程 | WebSocket |
| Agent 间通信 | 进程内通道（单体） → 消息队列（后续）|

### 决策 3：代码执行沙箱

**Docker 容器沙箱**，每个用户分配一个轻量容器，代码在其中编译运行。

### 决策 4：多模型调度策略

```
简单任务 → DeepSeek V3 (低成本)
复杂推理 → DeepSeek R1 (高质量)
向量检索 → BGE-M3 (本地部署)
降级策略 → 返回缓存/降级回答
```

## 五、架构原则

1. **KISS 原则** — 单体 → 微服务（按需拆分）
2. **数据驱动** — 每次学习/面试的数据都存储
3. **优雅降级** — LLM 调用失败 → 返回缓存内容
4. **关注 Token 成本** — 每个 Agent 调用记录消耗
5. **渐进式可观测** — 日志 → 指标 → 链路追踪
6. **国内优先** — 所有依赖确保国内可访问
